Lifecycle Models-Maturity Models

CMM(Capability Maturity Model)

CMM is a method of assessment.
Based on:

  • management styles using quantitative methods
  • generic process areas defines ‘what’ not ‘how’
  • five level capability model

CMM Levels

  1. Initial Level (chaotic)

    组织流程是非正式的、混乱的，甚至根本没有明确定义的流程

    • Level 1 managers & developers should not concentrate on metrics and their meanings
    • They should first attempt to adopt a process model

  2. Repeatable Process

    基本流程被确立，并有一定的原则规范

    • input: Problem statements or requirements specification

      output: Source code

    • Basic tracking established for cost, schedule and quality

    • Level 2 Metrics

      • Software size: Lines of code, Function points, classes or method counts
      • Personal efforts: person-months
      • Technical expertise
      • Employee turnover within project

  3. Defined Progress

    所有流程均被确立，并实现文字化、标准化，且互为融合

    • The software process for both management and engineering activities is documented and integrated into a standard process
    • Level 3 Metrics (in addition to metrics from lower maturity levels):
      • Requirements complexity:
        • Number of classes, methods, interfaces
      • Design complexity:
        • Number of subsystems, concurrency, platforms
      • Implementation complexity:
        • Number of code modules, code complexity
      • Testing complexity:
        • Number of paths to test, number of class interfaces to test
      • Thoroughness of testing:
        • Requirements defects discovered
        • Design defects discovered
        • Code defects discovered
        • Failure density per unit (subsystem, code module, class

  4. Managed Progress

    基于流程运作过程中的的数据信息，对流程及其质量进行测量

    • Detailed measures of the software process and product quality are collected.
    • Level 4 Metrics
      • Number of iterations per activity
      • Code reuse: Amount of producer reuse (time designated for reuse for future projects?)
      • Amount of component reuse (reuse of components from other projects and components)
      • Defect identification: How and when (which review) are defects discovered?
      • Defect density: When is testing complete?
      • Configuration management: Is it used during the development process? (Has impact on tracebility of changes).
      • Module completion time: Rate at which modules are completed (Slow rate indicates that the process needs to be improved).

  5. Optimizing Progress

    通过对流程表现的量化反馈，或试验性技术的前导运用，对流程进行不断改善提高

    • Continuous process improvement is enabled by quantitative feedback from the process and from piloting innovative ideas and technologies
    • Goal is to prevent the occurrence of defects

Internal Structure to Maturity Level

  • Except for level 1, each level is decomposed into key process areas (KPA)
  • Each KPA identifies a cluster of related activities that, when performed collectively, achieve a set of goals considered important for enhancing software capability.

Disadvantages of CMM

  • Organizations focus too much on next level as the target, rather than improving processes
  • CMM does not tell you how to achieve the goals, ‘what’ not ‘how’
  • No help to recover from an emergency situation

ISO/IEC 15504 (SPICE)

Software Process Improvement and Capability Determination

Processes of SPICE

Automotive SPICE

  • Automotive SPICE is a domain specific variant of the international standard ISO/IEC 15504 (SPICE).
  • Goal is to assess the maturity of suppliers of Electronic Control Units

Process capability level model (PCL-model)

Comparison of CMMI and SPICE

  • Basically, CMMI and SPICE are comparable in content and assessment procedures.
  • In some developments CMMI is broader than SPICE
    • integration of models
    • maturity levels
    • coverage of system development
    • different orders of magnitude of assessments
  • The engineering category is much more detailed in SPICE.
  • CMMI has more explanations.
  • SPICE has more process areas and practices.
  • CMMI widely used in the US; SPICE in Europe
